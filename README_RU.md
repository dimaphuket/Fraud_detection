# Обнаружение мошенничества с банковскими картами

Полноценный ML-проект по обнаружению мошеннических транзакций с кредитными картами на синтетическом датасете IBM (~24 миллиона транзакций, 1991–2019).

**Задача:** бинарная классификация — предсказать класс мошеннической операции
**Доступ к данным:** банк-эмитент (полный доступ к данным транзакций, карт и клиентов)  
**Дисбаланс классов:** ~0.12% мошеннических транзакций

---

## Результаты

| Модель | PR-AUC | ROC-AUC | Точность | Полнота | F1 |
|---|---|---|---|---|---|
| **XGBoost (настроенный)** | **0.883** | **0.9997** | **0.822** | **0.798** | **0.809** |
| XGBoost (базовый) | 0.860 | 0.9995 | 0.767 | 0.795 | 0.781 |
| CatBoost (настроенный) | 0.838 | 0.9989 | 0.779 | 0.760 | 0.770 |
| CatBoost (без настройки) | 0.828 | 0.9994 | 0.719 | 0.778 | 0.747 |
| Random Forest (ручная оптимизация) | 0.786 | 0.9993 | 0.758 | 0.715 | 0.736 |
| Стекинг (LogReg + RF) | 0.720 | 0.9992 | 0.171 | 0.973 | 0.291 |
| Логистическая регрессия (настроенная) | 0.581 | 0.9958 | 0.578 | 0.525 | 0.551 |
| Правиловая система оценки риска | 0.211 | 0.935 | 0.459 | 0.225 | 0.302 |
| Всегда предсказывать «нет мошенничества» | 0.001 | 0.500 | — | — | 0.000 |

> **Выбор метрики:** PR-AUC (площадь под кривой Точность–Полнота) является основной метрикой, поскольку ROC-AUC искусственно завышен при сильном дисбалансе классов и не отражает реальное качество обнаружения. Значения F1 вычислены при наилучшем пороге, найденном на валидационной выборке.

## Описание пайплайна

### 1. Подготовка данных (`1. Preliminary.ipynb`)
- Объединяет три исходные таблицы по ключам `User` / `CARD INDEX` (~24M строк).
- Применяет исправление ошибок датасета: заменяет некорректные значения `Merchant_State = "Italy"` путём выборки из реального распределения стран.
- Переименовывает столбцы и сжимает типы данных — **объём памяти снижен с 8.2 ГБ до 3.5 ГБ (~57%)**.
- Объединяет `Year`, `Month`, `Day`, `Time` в единый столбец `Timestamp`.
- Сохраняет во `fraud.parquet`.

### 2. Разведочный анализ данных (`2. EDA.ipynb`)
- Заполняет пропущенные значения: `Merchant_State` для онлайн-транзакций → `"ONLINE"`, создаёт бинарный флаг `Has_Error`, бинарный флаг `Is_Apartment`.
- Создаёт `Is_Online`, преобразует `Use_Chip` в бинарный признак.
- Выявляет и устраняет риски утечки данных из будущего: `PIN_Last_Changed_Year`, высококоррелированные столбцы.
- Проводит тесты статистической значимости (U-критерий Манна-Уитни для числовых, χ² для категориальных признаков).
- **Ключевой вывод:** мошенничество — это явление *на уровне торговца*; демографические данные держателя карты и место жительства статистически незначимы для предсказания фрода.
- Удаляет 8 малоинформативных / утекающих столбцов; сохраняет файл `fraud_after_EDA.parquet`.

### 3. Разработка признаков (`3. Feature_Engineering.ipynb`)
Признаки разделены на два типа:

**Строковые (без риска утечки):**
- `Amount_to_Income` = Сумма / (Годовой доход / 12)
- `user_age` — возраст клиента на момент каждой транзакции
- Признаки типов ошибок: `Error_Bad_CVV`, `Error_Bad_PIN`, `Error_Bad_Expiration`, `Error_Insufficient_Balance` и др.
- Временны́е признаки: `is_night`, `is_business_hours`, `is_weekend`, циклические кодировки (`hour_sin/cos`, `dow_sin/cos`)
- `amount_log`, `amount_round_10` (кратность $10 — частый паттерн мошенничества)

**Оконные / лаговые признаки на основе времени (вычислены строго по прошлым данным):**
- Показатели транзакций: `txn_count_5m_card`, `txn_count_1h_card`, `txn_count_5m_user`, `txn_count_1h_user`
- `time_since_prev_txn_card_min`, `time_since_prev_txn_user_min`
- Активность продваці: `merchant_txn_count_1h/24h`, `merchant_amount_sum_1h/24h`
- Исторические коэффициенты мошенничества: `merchant_fraud_rate`, `state_fraud_rate`
- `first_user_payment_to_this_merchant`, `state_changed_1d`, `card_burst_5m`, `errors_prev_1h`

**Анализ дрейфа (PSI / TVD по годам):**  
Анализ данных демонстрирует 2 эпохи с резко разным распределением признаков: до 2015 года и после → обучение ограничено периодом **2015–2019** (2020 исключён из-за аномально низкого уровня фрода). Сохраняются два итоговых датасета:
- `data_linear.parquet` — для логистической регрессии (сокращённый набор признаков, готов к StandardScaler)
- `data_tree.parquet` — для моделей на деревьях (готов к OrdinalEncoder, полный набор признаков)

### 4. Базовые не-ML модели и линейные модели (`4. Baseline_and_linear.ipynb`)
Пять не-ML модели позволили нам подобрать минимальный базовый уровень:

| Базовая линия | F1 | Описание |
|---|---|---|
| Всегда предсказывать 0 | 0.000 | Тривиальная |
| Случайный классификатор | 0.002 | На уровне базовой частоты фрода |
| Эвристика MCC + Сумма | 0.006 | Сумма > 3× среднего по MCC |
| Визуальная эвристика | 0.036 | ≥4 из топ-10 рискованных условий |
| **Скоринг на основе правил** | **0.302** | Банковские правила эпохи 1990-х |

Скоринг-система воспроизводит реальные антифродовые системы конца 1990-х и служит точкой отсчёта.

Логистическая регрессия обучается с кросс-валидацией `TimeSeriesSplit`; лучший результат: **PR-AUC = 0.581, F1 = 0.551**. Полиномиальные признаки и PCA результата не улучшают. Топ-20 признаков сохранены в `logreg_top_features.json`.

### 5. Модели на деревьях (`5. Tree_models.ipynb`)
- **Дерево решений (GridSearchCV):** PR-AUC = 0.498, F1 = 0.536
- **Случайный лес (ручная оптимизация):** `n_estimators=500`, `max_depth=25`, `min_samples_leaf=5`, `max_features=0.5` → **PR-AUC = 0.786, F1 = 0.736**
- Optuna (байесовский поиск) формально выигрывает на кросс-валидации, но переобучается; ручное ограничение глубины даёт лучшую обобщаемость.
- Случайный лес на топ-20 признаках логрег: PR-AUC = 0.639 — подтверждает, что логрег достиг своего потолка и мы приняли правильное решение сконцентироваться на древесных моделях.

### 6. Стекинг (`6. Stacking.ipynb`)
- OOF-стекинг: LogReg + RF в роли базовых моделей, LogReg в роли мета-модели.
- **Результат: неудача** — слишком большой разрыв в качестве между базовыми моделями. Мета-модель почти полностью игнорирует вклад LogReg, обеспечивая высокую полноту (0.973) ценой очень низкой точности (0.171), что неприемлемо на практике.

### 7. Бустинг (`7. Boosting.ipynb`)
- **XGBoost** (GPU `tree_method='hist'`, `scale_pos_weight`, `eval_metric='aucpr'`, ранняя остановка): **PR-AUC = 0.883, F1 = 0.809**
- **CatBoost** (GPU, `use_best_model=True`, ранняя остановка ~190 итераций): PR-AUC = 0.838, F1 = 0.747
- Гиперпараметры настроены с помощью `RandomizedSearchCV(TimeSeriesSplit)`.
- Лучшая модель сохранена как `best_xgb.pkl`.

**Бизнес-интерпретация финальной модели:** ~80% всего мошенничества корректно блокируется; ~82% отправленных оповещений являются реальным мошенничеством — модель пригодна для автоматической блокировки карт с управляемым объёмом ложных срабатываний.

### 8. Дополнительный эксперимент (`8. Bonus experiment.ipynb`)
Проверяется гипотеза: *мошенничество = нетипичный паттерн транзакций → мошеннические транзакции далеки от центроидов кластеров*.

Шесть независимых кластерных пространств (MiniBatchKMeans) обучаются на выборке 300К примеров:

| Пространство | Признаки | Кластеры |
|---|---|---|
| `transaction_profile` | Сумма, MCC, чип, время, география | 35 |
| `customer_profile` | Долг, FICO, кредитный лимит, доход, возраст | 20 |
| `behavior_profile` | Активность, время с прошлой транзакции, ошибки | 30 |
| `merchant_profile` | Активность и суммы по торговцу | 30 |
| `error_profile` | Флаги и счётчики ошибок | 10 |
| `risk_context_profile` | Исторические коэффициенты фрода по торговцам и штатам | 20 |

Номера кластеров + расстояния до всех центроидов (~145 признаков) передаются в LogReg и XGBoost.

**Вывод:** кластеризация успешно выделяет поведенческие архетипы, расстояния до центроидов несут предсказательный сигнал о мошенничестве. Однако результативность (PR-AUC ~0.65–0.70) не приближается к полнопризнаковой древесной модели. Кластерные признаки могут быть полезны как дополнительные признаки при обучении древесной модели.

## Ключевые выводы

1. **Основные предикторы мошенничества:** `Is_Online`, `Use_Chip`, `Has_Error`, `MCC`, география торговца (штат/город), время транзакции, признаки активности, исторические коэффициенты фрода.
2. **Не являются предикторами:** демографические данные держателя карты (пол, адрес, город/индекс), бренд и тип карты.
3. **Дрейф данных:** для обучения пригодны только данные 2015–2019; более ранние годы снижают качество модели.
4. **Разработка признаков — главный рычаг роста**: признаки активности, исторические коэффициенты фрода и детектирование всплесков активности обеспечивают большую часть разрыва между LogReg (PR-AUC 0.58) и XGBoost (PR-AUC 0.88).
5. **Стекинг не работает** при слишком большом разрыве в качестве базовых моделей.
6. **Кластеризация как разработка признаков** сохраняет полезный сигнал, но не может заменить вручную созданные признаки.

## Воспроизведение результатов

> Исходные CSV-файлы не включены в репозиторий из-за их размера. Скачайте датасет с [IBM Credit Card Transactions](https://www.kaggle.com/datasets/ealtman2019/credit-card-transactions/data) и поместите файлы в папку `data/`.

**Установка зависимостей:**
```bash
pip install -r requirements.txt
```

**Запуск ноутбуков по порядку:**
```
1. Preliminary.ipynb
2. EDA.ipynb
3. Feature_Engineering.ipynb
4. Baseline_and_linear.ipynb
5. Tree_models.ipynb
6. Stacking.ipynb
7. Boosting.ipynb
```
Ноутбук 8 содержит не обязательный эксперимент.

**Аппаратное обеспечение:** ноутбуки XGBoost и CatBoost используют GPU через CUDA (`device="cuda"`). Для запуска на CPU замените `device="cuda"` → `device="cpu"` и `task_type="GPU"` → `task_type="CPU"` в соответствующих ячейках. Внимание! Это существенно замедлит работу!
